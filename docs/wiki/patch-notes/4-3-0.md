# 4.3.0 — Sentinel adversarial assurance on the hardened 4.2.1 base

This release adds GUILD-039, the Sentinel Adversarial Assurance Guild, and SO-026
continuous artifact security monitoring, as an **additive assurance layer** on top of
the fully-remediated LPOS v4.2.1 base. Sentinel does not change the security posture
established by 4.2.1; it adds an independently reviewed adversarial-assurance capability
over Chip-created work.

## Additive assurance — what Sentinel is and is not

- Sentinel is **additive**. The 14 findings from the external audit were closed by the
  v4.2.1 remediation (verified-channel approvals, the hash-chained event store, the
  hardened dashboard and monitor, the truthful compliance readiness monitor, the
  always-on schema gate, and the signing/SBOM tooling). **Sentinel did not close those
  findings**, and its presence must never be represented as doing so.
- Sentinel is **not trusted because it is a security guild**. Its raw scanner output
  starts explicitly untrusted and must pass LPOS's ordinary fresh-context independent
  adversarial review plus deterministic structural verification before any finding can
  block work, authorize anything, or be presented to the Principal as fact.

## Added

- Constitution Article VIII: no component or new guild is trusted by designation, and no
  producer may approve, certify, remediate, or close its own material output.
- SPECIALIST-033, Adversarial Assurance Engineer, and SO-026, Continuous Adversarial
  Assurance (scheduled backstop every five minutes).
- Passive security checks on every persisted Chip artifact revision, with hash-bound
  untrusted raw assessments, fresh-context independent review, and deterministic
  structural re-scanning.
- Principal security reports with prioritized remediation and verification steps.
  Reviewed Critical/High findings block completion of only the affected revision, and
  only after the assessment itself passes independent review.
- `lpos sentinel` administration commands (`status`, `scan`, `reports`, `show`, `ack`).

## Improvements over the development candidate

Two latent weaknesses in the Sentinel candidate were fixed during integration onto the
hardened base:

- **Sentinel's own record tables are anchored to the tamper-evident event chain.** In
  addition to append-only triggers, every assessment, review, and report hash is
  recorded in the 4.2.1 SHA-256 hash-chained event stream, and `integrity_check` now
  reconciles the Sentinel tables against that chain. Editing a stored record after
  dropping its triggers — or flipping a review's trust decision — is detected.
- **The active-engagement authorization gate goes through the 4.2.1 verified-channel
  assertion path.** A fabricated `MessageIdentity` whose provider has no registered
  channel verifier cannot mint an approval grant, so it cannot authorize a Sentinel
  active engagement. The gate fails closed.

## Counts

## Safety and scope

Sentinel is passive by default: continuous operation never executes an artifact, opens a
network connection, uses credentials, invokes a shell, or changes live state. Active
penetration testing requires a separately approved, isolated, time-bounded scope bound to
an ordinary LPOS approved action. Sentinel is an additive defense layer, not a proof that
work is secure, not a substitute for an authorized human-led penetration test, and not a
SOC 2 attestation.

## Hardening and governance in this release

- **LPOS-15 (audit follow-up) closed.** Monitor and compliance state files are now created 0600 via the shared secure-write helpers (regression-tested under umask 022), and `lpos doctor --hermes-root <root>` audits state-file permissions and fails closed if any are group/other-readable — matching the coverage the 4.2.1 notes claimed.
- **OS Upgrade Radar captured (not installed).** `docs/OS-UPGRADE-RADAR.md` records the external candidates from the 2026-07-23 architect handoff as governed design inputs behind an acceptance gate; none are installed — promotion requires its own build, audit, and Principal approval.
