# 4.2.1

A security and assurance-integrity patch: remediation of the 14-finding external audit
of v4.2.0. Every code-level finding is fixed with an adversarial regression test
mirroring the audit's own reproduction.

## What changed

- **Truthful self-assessment (LPOS-01, LPOS-02 — Critical/High).** The compliance
  module is now a *control readiness monitor*: it never emits "compliant" or
  "effective" about itself. Fresh installs report `not_assessed` / `insufficient_evidence`;
  distinct audit runs and days are counted honestly (21 controls in one execution is
  one run); statuses carry `attestation: false`, `issued_by_cpa: false`; structural
  checks are labeled as structural evidence and organizational criteria are listed as
  requiring human evidence.
- **Monitor hardening (LPOS-03, LPOS-04 — Critical/High).** No shell execution
  anywhere; executable checks come only from the admin-owned `monitor/approved-checks.json`
  — agent-registered services can no longer define commands or receive credentials.
  Tokens are bound to their connector's fixed origin, token files must live in the
  monitor secrets directory, private/loopback/metadata addresses are refused for
  agent-supplied URLs, and redirects are refused.
- **Verified approval identity (LPOS-05 — High).** Approvals now require an assertion
  minted by a registered channel verifier; a fabricated `MessageIdentity` with an
  allowlisted sender string is rejected. Live consequential adapters require a real
  provider-signature verifier and a verified round trip.
- **Dashboard boundary (LPOS-06, LPOS-09 — High/Medium).** Session-token auth on every
  API route, Host validation (kills DNS rebinding), Origin checks, JSON-only capped
  bodies, security headers, generic errors, mandatory loopback; the scanner no longer
  follows symlinks out of approved roots and out-of-root kanban paths are dropped.
- **State confidentiality (LPOS-07 — High).** Databases and state files are created
  0600 in 0700 directories regardless of umask, with startup permission repair.
- **Tamper-evident evidence (LPOS-08 — High).** The event store and the compliance
  history are hash-chained with verification (`lpos doctor`, `python -m
  lpos_engine.compliance verify`); the audit's row-removal and trigger-drop
  reproductions are now detected. Optional off-host checkpoint keys strengthen this.
- **Subprocess boundary (LPOS-10 — Medium).** Streaming output caps (bounded peak
  memory), process-group kill, allowlist environment, resource limits, executable
  pinning — documented honestly as privilege reduction, not a sandbox.
- **Release authenticity (LPOS-11 — Medium).** Ed25519 release signing
  (`tools/sign_release.py`, verified automatically by `verify_release.py` when
  present) and a CycloneDX SBOM (`tools/make_sbom.py`).
- **Always-on schema gate (LPOS-12 — Medium).** A built-in structural meta-validator
  runs in every install; a meta-invalid schema now fails installation instead of
  silently downgrading to JSON parsing.
- **Docs accuracy (LPOS-14 — Low)** and **SOC 2 program scaffolding (LPOS-13)**:
  `docs/SOC2-PROGRAM.md` defines the organizational half a Type 2 examination needs.

## What you will notice

Dashboard API calls now need the session token from `dashboard/token`. Monitor
entries that carried their own commands now report `unknown` until an admin approves
a check template. The compliance page is retitled "Control Readiness Report" and a
fresh system honestly shows insufficient evidence instead of green.
